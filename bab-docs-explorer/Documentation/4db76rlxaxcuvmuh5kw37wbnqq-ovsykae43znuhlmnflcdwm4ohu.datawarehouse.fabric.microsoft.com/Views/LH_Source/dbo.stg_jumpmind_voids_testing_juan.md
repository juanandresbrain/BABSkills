/* =============================================================================    stg_jumpmind_voids.sql — VOID → POST_VOID Enrichment Subroutine    =============================================================================    Purpose: Replicates the non-trivial VOID enrichment logic in Stage A of the             JumpMind C# pipeline before any void_flag computation downstream.     Source: BBW_C_Sharp_JumpMind/JumpMindSATService/Service1.cs lines 441-501     Logic (literal from C#):      When `transTypeCode = 'VOID'` AND `voidedDeviceId` is non-null AND           `voidedSequenceNumber` is non-null:        1. Look up the original transaction by store/register/sequence from           jumpmind_sls_trans (or the legacy JumpMindAPI_Logging table)        2. Copy the original's content into the void record        3. Override fields: `sequenceNumber`, `deviceId` (=register), `createTime`,           `barcode`, `tillId`, `username`        4. Relabel `transTypeCode` as `POST_VOID`        5. Negate all tender amounts: payment.tenderAmount = -1 × original        6. Rewrite price modifier sequenceNumbers to match the new transaction     This view emits one row per enriched VOID (now relabeled POST_VOID) — to    be UNION-ALL'd into stg_jumpmind_transactions for non-VOID rows.     Source-unification CTE order (per build prompt):      1. raw_voids — VOIDs to be enriched      2. originals — looked up by (store, register, voided_sequence)      3. enriched_voids — composed output      (No derivation CTEs in this view — derivation happens downstream in       stg_jumpmind_lines / stg_jumpmind_transactions)     Schema mirrors `jumpmind_sls_trans` plus a `void_enriched_flag` column to    indicate this row is a Stage-A-enriched POST_VOID rather than a raw record.     ⚠ DEPENDS ON: LH_Source has the VOIDed originals retained.
